# drskill v0.1 design

Date: 2026-07-19
Status: approved
Parent document: `initial_design_doc.md` (Draft v1)

## What v0.1 is

v0.1 is the useful skeleton described in section 10 of the design document. It is a read only command line tool that finds the coding agents installed on a machine or configured in a repo, works out the exact set of skills each agent loads, runs the Tier 1 checks over those sets, counts tokens, and prints a report in which every finding ends in a fix command or an ack command. Decisions are recorded in a committed `drskill.toml` ledger. The tool works with zero configuration and makes zero LLM calls.

## What v0.1 is not

- No installer. Fix commands delegate to `npx skills` and `gh skill`.
- No Tier 2 heuristics, Tier 3 injection checks, or the LLM tier. Those are v0.2 and v0.3.
- No `explain`, `share`, `cache`, or `telemetry` commands.
- No MCP support.
- No writes to `skills-lock.json`, ever.

## Architecture

The code is a pipeline with four stages. Each stage has one job and a defined output, and the stages only communicate through those outputs.

1. Discovery. Detect installed harnesses and walk each one's search paths. The output is a list of raw skill instances, where each instance is a path plus the harness and scope it was found under.
2. Resolution. Turn raw instances into per harness effective sets of `Contributor` objects. This stage resolves symlinks, deduplicates by real path, applies each harness's precedence rules, and parses frontmatter and bodies.
3. Checks. Each check is a function that takes the resolved world and returns a list of `Finding` objects. Checks are registered in a flat registry keyed by check id. The ledger filter then removes findings whose fingerprint matches a current ack.
4. Reporting. One renderer prints the human report with rich. A second emitter prints the same findings as JSON. Both read the same `Finding` list.

Tier 2, Tier 3, and Tier 4 checks are added later as new entries in the check registry. MCP support is added later as a second discovery provider. Neither change touches the middle of the pipeline.

## Package layout

```
pyproject.toml            # uv managed, Python >= 3.11
src/drskill/
    __init__.py
    cli.py                # typer app, all commands
    models.py             # Contributor, Finding, Provenance, Deployment, TokenCost
    harnesses.py          # harness definition schema and loading
    data/harnesses.toml   # harness definition table (see below)
    discovery.py          # stage 1
    resolution.py         # stage 2
    checks/
        __init__.py       # registry
        shadowing.py      # name-shadow, double-load
        duplicates.py     # exact-duplicate, near-duplicate, MinHash
        spec.py           # spec violation checks
        filesystem.py     # broken-symlink
        lockfile.py       # lockfile-drift
        budget.py         # token budget checks
    ledger.py             # drskill.toml read and write, ack matching
    tokens.py             # tiktoken behind a lazy import
    report.py             # rich renderer and JSON emitter
tests/
```

Dependencies on the default path: typer, rich, pydantic, pyyaml, tomli-w, tiktoken. `tomllib` comes from the standard library. MinHash is hand rolled in `duplicates.py`, roughly 40 lines, no dependency.

## Data models

The models follow section 4 of the design document. All are pydantic models.

```python
class Deployment(BaseModel):
    harness: str          # harness id, e.g. "claude-code"
    path: Path            # the path the harness reads
    via_symlink: bool
    shadowed_by: str | None = None   # contributor id of the winner, when shadowed

class Provenance(BaseModel):
    kind: Literal["skills-lock", "gh-skill", "unmanaged"]
    source: str | None    # repo url or lock entry name when known

class TokenCost(BaseModel):
    catalog_tokens: int   # name + description, approximate
    body_tokens: int      # full body, approximate

class Contributor(BaseModel):
    id: str               # stable identity derived from real path
    kind: Literal["skill"]
    name: str             # frontmatter name
    source: Provenance
    scope: Literal["project", "user"]
    deployments: list[Deployment]
    routing_text: str     # the description
    body: str
    token_cost: TokenCost
    content_hash: str     # sha256 of normalized content

class Finding(BaseModel):
    check_id: str
    severity: Literal["error", "warning"]
    contributors: list[str]      # contributor ids
    contributor_names: list[str] # skill names, for the report and ack matching
    harnesses: list[str]
    message: str
    fix_commands: list[str]
    fingerprint: str
```

The `kind` field on `Contributor` only allows `"skill"` in v0.1. The literal grows when MCP providers arrive.

## Harness definitions

Harness knowledge lives in a data file, `data/harnesses.toml`, not in code. Each entry has these fields:

- `id` and `display_name`.
- `detect`: markers that show the harness is present, e.g. a config directory in the project or home directory.
- `project_paths` and `global_paths`: ordered lists of skill directories.
- `search_order`: either `"project-first"` or `"global-first"`. Both of the precedence rules this document first sketched (`"path-order"` and `"first-found"`) reduce to one mechanism: the first skill found across the combined ordered path list wins, and `search_order` decides whether project paths or global paths come first in that list. A harness that does not shadow on name collisions at all (Codex behaves this way) cannot be expressed yet; supporting a `"none"` value is a recorded follow-up, and Codex stays best effort until then.
- `recursive`: whether the harness finds `SKILL.md` files at any depth.
- `root_md_paths`: paths where a bare `.md` file in the directory root counts as a skill.
- `verified`: whether we have confirmed these rules against the harness's own docs or source.

The target verified set for v0.1 was Claude Code, Cursor, Codex, Copilot, Gemini CLI, and Pi, plus the universal `.agents/skills` directory. The implementation outcome (recorded 2026-07-19): Claude Code, Pi, and Gemini CLI are verified against their docs or source. Codex, Cursor, and Copilot stay `verified = false` with their research recorded in comments. Codex because its no-shadowing collision behavior does not fit the `search_order` model yet, Cursor and Copilot because their docs do not state which path wins on a name collision. Pi's verified rules:

- Global paths: `~/.pi/agent/skills/`, then `~/.agents/skills/`.
- Project paths: `.pi/skills/`, then `.agents/skills/` in the working directory and its ancestors.
- Precedence is first found. Directories with a `SKILL.md` are found recursively. A bare `.md` file counts as a skill only in `~/.pi/agent/skills/` and `.pi/skills/`.

All other harness paths are seeded from the agent directory mapping in `vercel-labs/skills` (MIT, vendored, upstream commit recorded in the data file header; 66 entries as shipped). A harness is marked verified only when its rules are confirmed by hand against that harness's own docs or source. Harnesses that appear only in the vendored table stay `verified = false`. The tool still detects them, but the report labels their findings as best effort.

## Discovery and resolution

Discovery runs per harness. For each detected harness the tool walks its search paths in order and records every skill instance: the path, the scope, whether the path is a symlink, and the position of its parent directory in the precedence order. In project mode the tool walks project paths and global paths, because both feed the harness's effective set. With `--global` the tool only walks global paths.

Resolution then builds the effective set:

1. Resolve every instance path to its real path.
2. Group instances by real path. Each group becomes one logical `Contributor` with one deployment per instance. A skill symlinked into five agent directories is one contributor with five deployments.
3. For each harness, apply its precedence rule to name collisions among the contributors it loads. The loser instances stay in the model, and each losing deployment records the winner in its `shadowed_by` field. The `name-shadow` check reads this.
4. If one harness reaches the same logical skill through two different physical instances, meaning two copies with equal normalized content hashes rather than one real path, that is not deduplicated. The `double-load` check reads this.
5. Parse frontmatter with pyyaml and split the body. A file whose frontmatter does not parse still becomes a contributor, with a flag the spec checks read.

Provenance is read from two sources. `skills-lock.json` is the manifest of record and is never written. `gh skill` provenance is read from SKILL.md frontmatter keys. Everything else is classified unmanaged.

Normalized content is defined as the SKILL.md text with the `gh skill` provenance frontmatter keys removed and line endings normalized to `\n`. The sha256 of this is `content_hash`. This means the same skill installed through different tools hashes the same, and a metadata only update does not disturb an ack.

## Checks

| check id | severity | fires when |
|---|---|---|
| `name-shadow` | warning | Two skills share a name in one harness's set and one shadows the other. The message names the winner and the rule that picked it. |
| `double-load` | error | One harness loads the same logical skill twice through two directories. |
| `exact-duplicate` | warning | Two contributors have equal normalized content hashes under different names or paths. |
| `near-duplicate` | warning | Jaccard similarity of MinHash signatures over word shingles is at or above the threshold. The default threshold is 0.85 and can be changed in the ledger. |
| `spec-name-mismatch` | error | Frontmatter `name` does not match the folder name. |
| `spec-missing-description` | error | The description is absent or empty. |
| `spec-description-too-long` | error | The description exceeds 1024 characters. |
| `spec-invalid-frontmatter` | error | The frontmatter does not parse as YAML. |
| `frontmatter-angle-brackets` | warning | Frontmatter values contain angle brackets, which the spec flags as an injection vector. |
| `broken-symlink` | error | A symlink in a skill directory points at nothing. |
| `lockfile-drift` | warning | A skill's tree hash does not match its `skills-lock.json` entry, or a locked skill is missing on disk. The mismatch message attributes the likely cause, e.g. a `gh skill update` or a hand edit, and does not call it corruption. Hash comparison is self-calibrating (added during implementation): when every hashed entry mismatches, the lockfile's hash algorithm evidently is not one drskill can reproduce, so the tool emits one aggregate "cannot verify" warning instead of accusing every skill. Mixed results mean the algorithm lines up, so per-skill mismatches are reported. |
| `budget-catalog-tokens` | warning | A harness's total catalog tokens exceed `[budget] catalog_tokens_max`. |
| `budget-body-tokens` | warning | A skill's body tokens exceed `[budget] body_tokens_warn`. |

Every finding ends in either fix commands or an ack command. Fix commands delegate to the tool that owns the operation, e.g. `npx skills remove <name>` or `gh skill update <name>`. The ack line is printed exactly as the user should type it.

## Token accounting

Counting uses tiktoken with the `o200k_base` encoding. The import is lazy so the CLI starts fast. All counts are labeled approximate in the report, because each harness renders its catalog differently. `catalog_tokens` counts the name plus the description, which is what loads at session start. `body_tokens` counts the full body. Budget defaults are 6000 catalog tokens per harness and 20000 body tokens per skill, matching the design document.

## Ledger

`drskill.toml` sits at the repo root and is committed. It holds configuration and decisions:

```toml
[budget]
catalog_tokens_max = 6000
body_tokens_warn   = 20000

[thresholds]
near_duplicate = 0.85

[[ack]]
check = "near-duplicate"
skills = ["docx-report", "documentation-writer"]
fingerprint = "sha256:..."
note = "docx is output format specific; keeping both"
date = 2026-07-19
```

A finding's fingerprint is the sha256 of the check id plus the sorted content hashes of the involved contributors. An ack whose fingerprint matches a current finding silences it. When any involved skill's normalized content changes, the fingerprint no longer matches and the finding returns.

`drskill ack <check-id> <skill>...` finds the matching current finding, computes its fingerprint, and appends an ack entry. Skill names are optional for findings that involve no specific skill, e.g. the aggregate lockfile warning: `drskill ack <check-id>` alone matches the unique finding with that check id and no contributor names. The file is written with tomli-w. Comments in the file are not preserved on write. This is an accepted v0.1 limitation and is documented in the README.

In global mode the ledger is `~/.drskill.toml`.

## CLI

```
drskill scan [--global] [--ci] [--json]
drskill ack <check-id> <skill>... [--note "..."]
drskill list [--tokens] [--harness <id>] [--global]
drskill init
```

- `scan` runs the pipeline and prints the report. `--json` prints the `Finding` list as JSON instead.
- `ack` records a decision as described above.
- `list` prints each harness's effective set. `--tokens` adds per skill token columns and per harness totals.
- `init` writes a starter `drskill.toml` with the default budgets and thresholds. It refuses to overwrite an existing file.

Exit codes: 0 when clean or when everything is acknowledged, 1 when errors are present, 2 when unacknowledged warnings are present and `--ci` was passed. Without `--ci`, warnings alone exit 0.

The report groups findings by severity. Each entry names the harnesses it affects and ends in its fix commands or its ack line. Findings from unverified harnesses carry a best effort label.

## Testing

- Unit tests with pytest. Synthetic harness trees are built in `tmp_path` fixtures, including symlink farms, `gh skill` style copies, broken symlinks, and shadowing setups.
- `--json` output is covered by structural assertions on the parsed JSON (implemented in place of the golden file tests this document first proposed; same coverage, less brittle).
- Ledger round trips are tested: scan, ack, scan again silent, edit the skill, scan again and the finding returns.
- A smoke test runs `scan` against this repo's own `.claude` and `.agents` directories.

### Conformance suite

Check behavior on skill content is tested through a data driven conformance suite, not through hand written assertions per case.

- Each case is a directory under `tests/conformance/cases/<case-name>/` holding a small fixture skill tree.
- Next to the tree, an `expect.toml` file states the expected outcome as data. `[[expect]]` entries list a check id and the skill names that must fire it. `[[forbid]]` entries list pairs that must not fire it, which guards against false positives.
- One parametrized pytest runner loads every case, runs the scanner over the fixture tree, and asserts that every expected finding appears and every forbidden finding does not.

Adding a case means writing skill files and stating the clash. No Python is needed. The case format is versioned because later milestones reuse it. The v0.2 threshold tuning work turns the mega repo's known duplicate groups into cases in this format, and the v0.3 DSPy compilation reads the same labeled pairs with one added verdict field.

Corpus tuning against the community mega repo is a v0.2 activity, because it belongs with the Tier 2 threshold work.

## Implementation order

1. Project scaffolding, models, and the harness data file with Pi and Claude Code verified.
2. Discovery and resolution with tests.
3. Checks, in the order spec violations, filesystem, duplicates, shadowing, lockfile, budgets.
4. Ledger and ack.
5. Report, JSON output, exit codes, `init`.
6. Verify the remaining core harnesses and finish the data file.
